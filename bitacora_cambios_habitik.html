1. Resumen Ejecutivo y Alcance del Proyecto

El presente documento unifica en un único registro maestro el reporte técnico inicial del Sprint 1 y la bitácora de ingeniería de las últimas sesiones de estabilización. El proyecto Habitik articula una solución frontend nativa e híbrida (Flutter) comunicada mediante tokens JWT contra una API REST en Node.js/TypeScript y una base de datos PostgreSQL alojada en Railway Cloud.

Principio Arquitectónico Rector El diseño del sistema prioriza la autonomía del cliente móvil. La totalidad de las correcciones visuales, de sesión y de hardware fueron resueltas en lib/. Únicamente tras presentarse una encrucijada de escalabilidad de datos en múltiples estaciones de trabajo, el usuario autorizó explícitamente una excepción quirúrgica en el servidor para cruzar tablas relacionales.

3. Catálogo Técnico de Peticiones HTTP (GET / POST / PATCH) y El Por Qué

La comunicación entre la aplicación cliente en Flutter y los servicios en la nube de Railway se rige estrictamente por el protocolo HTTP sobre TLS (HTTPS). A través de la clase centralizada ApiClient apuntando al host oficial https://backendhabitik-production.up.railway.app, estructuramos las siguientes peticiones:

Método y Endpoint Rol Permetido Payload / Parámetros Por qué se utiliza (Justificación Técnica)
POST /auth/register Público (Anónimo) email, password, nombre, nombreFamilia Creación Atómica: Ejecuta una transacción SQL en servidor que crea al usuario, le genera su perfil oficial y aprovisiona el registro maestro del Hogar Familiar en un solo paso.
POST /auth/login Público (Anónimo) email, password Emisión de Identidad: Valida credenciales contra bcrypt, retorna un token firmado token_jwt (24h) y entrega el objeto profile enriquecido dinámicamente con el nombre del hogar.
POST /onboarding Jefe de Familia (Admin) tipoCalefaccion, habitaciones, personas, electrodomesticos[] Persistencia de Perfil Energético: Guarda las respuestas de infraestructura del hogar. Su presencia en la BD activa la bandera de finalización para romper bucles infinitos en cliente.
GET /familia/invite Jefe de Familia (Admin) Cabecera: Authorization: Bearer <JWT> Generación Criptográfica QR: Solicita al servidor un UUID temporal (invite_token) que expira en 10 minutos. Es el string vital que dibuja el recuadro QR gigante de 200x200 px.
POST /familia/join Miembro Autenticado invite_token, user_id Emparejamiento de Hogar: Disparado al escanear el QR o pegar el UUID con el botón de portapapeles. Vincula el family_id al Miembro en PostgreSQL y gatilla re-login silencioso.
PATCH /familia/nombre Jefe de Familia (Admin) nombre Normalización de Hogar: Permite corregir o actualizar el nombre visual de la familia en la tabla families tras haber superado las fases de inicialización.
Por qué estandarizamos cabeceras en ApiClient.dart Para evitar inyección de código e inconsistencias de serialización, cada solicitud saliente inyecta automáticamente los encabezados estándar: Content-Type: application/json y Accept: application/json, anexando el JWT activo cuando el usuario ya se encuentra dentro de sesión.

4. Resolución de Bucles de Onboarding y Caché de Sesión

Al probar el ingreso con cuentas reales creadas en producción (como benjafront@habitik.com y matiasfront@habitik.com), el sistema atrapaba al usuario en un bucle infinito que lo regresaba a la pantalla de bienvenida.

Solución Implementada en Cliente:

  • Evaluación Dinámica de Respuestas: Al deserializar perfiles (UserProfile.fromJson), el código ahora inspecciona si el atributo onboarding_answers contiene respuestas en la base de datos. De ser positivo, marca automáticamente onboardingCompleted = true y salta directo al Dashboard.
  • Caché Inteligente de Identidad: Para cuentas de Miembros, la API no siempre adjunta el nombre del hogar en texto plano. Se creó un almacenamiento en SessionService que asocia en SharedPreferences el UUID de familia con su nombre visual.
  • Cierre de Sesión Limpio: El método clearSession() fue refactorizado para eliminar únicamente las credenciales temporales y el JWT, preservando las configuraciones nativas del dispositivo.

5. Pipeline de Sincronización en la Nube y Autenticación Silenciosa

Para vincular en tiempo real a un integrante recién invitado con el hogar del Jefe de Familia sin requerir un reinicio manual de la aplicación, se orquestó un flujo en tres etapas:

Etapa Acción en Cliente Flutter Endpoint Involucrado Efecto Persistente en Base de Datos
Paso 1 Escaneo de QR o pegado de token UUID. POST /familia/join Vincula el family_id al perfil del usuario en la tabla profiles.
Paso 2 Re-login silencioso en segundo plano. POST /auth/login Refresca el token JWT otorgando permisos oficiales del rol miembro.
Paso 3 Persistencia permanente en memoria local. SharedPreferences Memoriza en disco el family_name para futuras recargas de la app.

6.2 Resolución del Misterio: "DEVS" vs "Hogar Familiar" al clonar proyecto

Fenómeno detectado: En la PC principal del proyecto, iniciar sesión mostraba correctamente el título oficial (Hogar: DEVS). Sin embargo, cuando otro compañero clonó el repositorio en una PC virgen e inició sesión, su interfaz mostraba Hogar: Hogar Familiar.

Diagnóstico Arquitectónico: El endpoint POST /auth/login ejecutaba una lectura plana sobre profiles: 

SELECT * FROM public.profiles WHERE id = $1

La tabla profiles almacena el entero relacional (family_id), pero no almacena el texto del nombre (que vive en families). En la PC principal, SharedPreferences ya retenía el caché "DEVS". En la PC limpia, al recibir family_name: null, el cliente recurría al texto por defecto.

Excepción Aprobada a Regla 1 Para evitar fijar textos artificiales en Flutter y permitir que la app escale dinámicamente con miles de familias reales, el usuario otorgó autorización explícita para intervenir la capa SQL del servidor.

Implementación de Cruce SQL Relacional: Se reemplazó la consulta básica por un LEFT JOIN: 

// Archivo: backend/src/repositories/userRepository.ts
async getProfileById(id: string): Promise {
  const sql = `
    SELECT p.*, f.nombre AS family_name, f.family_code 
    FROM public.profiles p 
    LEFT JOIN public.families f ON p.family_id = f.id 
    WHERE p.id = $1
  `;
  const res = await query(sql, [id]);
  return res.rows[0];
}

Tras recompilar automáticamente en Railway Cloud, cualquier estación de trabajo virgen que clone el repositorio hoy extrae el nombre oficial directamente desde PostgreSQL.

7. Auditoría y Consultas SQL de Verificación (DBeaver)

Para inspecciones directas sobre la base de datos de producción en Railway Cloud, se establecieron las siguientes consultas:

A) Control general de cuentas, roles y respuestas guardadas: 
SELECT u.email, p.nombre, p.rol, p.family_id, p.onboarding_answers, u.created_at
FROM public.users u
JOIN public.profiles p ON u.id = p.id
ORDER BY u.created_at DESC;
B) Auditoría relacional de emparejamiento por Hogar: 
SELECT p.nombre AS integrante, p.rol, f.nombre AS hogar_familiar, f.family_code
FROM public.profiles p
JOIN public.families f ON p.family_id = f.id
ORDER BY f.created_at DESC, p.rol;
