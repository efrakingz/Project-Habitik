📁 Estructura de Directorios (backend/src/)

El backend está construido bajo el patrón de arquitectura por capas (Controladores, Servicios, Repositorios) garantizando modularidad y separación de responsabilidades:

src/
config/ → db.ts (Conexión al pool de PostgreSQL en Railway)
controllers/ → Manejadores de peticiones HTTP, validaciones básicas de inputs
middleware/ → auth.ts (Validación de token JWT y control de acceso del Jefe/Admin)
models/ → types.ts (Tipos de datos interfaces en TypeScript)
repositories/ → Capa de persistencia directa, ejecuta queries SQL
routes/ → Definición de los paths URL y asignación de middlewares
services/ → Capa de reglas de negocio y cálculos del sistema
server.ts → Archivo de arranque, configuración de Express y Healthcheck

⚡ Flujo de una Petición

1. Cliente (Flutter / HTTP Client)
Envía petición con Header: Authorization Bearer JWT
↓
2. Rutas & Middleware (Routes & verifyToken)
Extrae y decodifica el token. Verifica expiración y roles.
↓
3. Controlador (Controller)
Valida los datos de entrada (body) y responde con estados HTTP correctos.
↓
4. Servicio (Service)
Calcula fórmulas, ejecuta algoritmos anti-trampas e impone restricciones.
↓
5. Repositorio (Repository)
Se comunica con la base de datos a través de consultas parametrizadas seguras.

🔗 Endpoints y Reglas de la API

El backend expone 5 endpoints principales autenticados y públicos con lógica específica desarrollada por tus compañeros:

POST /auth/register
Público
Registra un nuevo Jefe de Familia y crea su Hogar simultáneamente. Todo el proceso está envuelto en una transacción de base de datos (si algo falla, nada se crea). Genera un código de hogar aleatorio de 6 dígitos únicos y retorna un token JWT válido por 24 horas.

{
  "email": "jefe@habitik.cl",
  "password": "mi_pass_seguro_123",
  "nombre": "Carlos",
  "nombreFamilia": "Familia López"
}
JSON RESPONSE (201 Created)
{
  "user_id": "8b94df80-e882-4f36-81cf-309d9435b801",
  "family_id": "cf7a5f6e-fa5e-4c7b-891d-55e1db9ef444",
  "token_jwt": "eyJhbGciOiJIUzI1NiIsInR..."
}

POST /reto/ducha
Cualquier Usuario Autenticado
Registra una sesión de ducha cronometrada. 🛡️ Regla anti-trampas: Si la ducha registrada dura menos de 180 segundos (3 minutos), se almacena en la base de datos pero se marca como invalida (retorna valido: false en formato JSON con código de respuesta HTTP 200). 🧮 Algoritmo de Cálculo de Baseline

¿Cómo funciona?

El archivo onboardingService.ts estima un consumo de referencia mensual dependiendo del rol del usuario. Hay dos tipos de estimaciones:

1. Por Infraestructura (Rol: Jefe / Admin)

Estima la base total del hogar sumando el número de personas en el hogar, la cantidad de habitaciones, el tipo de calefacción (la eléctrica añade una cantidad alta) y los electrodomésticos activos (como secadoras).

2. Por Hábitos Personales (Rol: Miembro)

Estima la porción del miembro según su tiempo en la ducha promedio, las horas que pasa frente a pantallas y añade un "Crédito Verde" de reducción de luz si recicla activamente.

Fórmula Luz (Jefe de Familia)
BaselineLuz (kWh) = 120 + (habitaciones * 30) + (personas * 20)
+ 150 (si calefacción es eléctrica)
+ 20 (si calefacción es gas)
+ (electrodomésticos * 15)
Fórmula Agua (Miembro)
Duchas al Mes = tiempoDucha * 10 litros/min * 30 días
BaselineAgua (m³) = Duchas al Mes / 1000
Costos Financieros Estimados
Costo Luz = kWh × $150 CLP
Costo Agua = m³ × $1,200 CLP
